Everything the agent does is persisted; the **governance read-model** is the read/analytics side of that data — cost and usage rollups, the run lifecycle, per-tool stats, run reliability, and a cross-thread approvals inbox. It lives behind one interface, `AgentGovernanceQueries`, so the [dashboard SPA](/docs/agent/governance/dashboard) and the [Telescope tab](/docs/agent/governance/telescope) consume the same rolled-up numbers.

<Callout type="info" title="Separate from the store on purpose">
`AgentStore` owns the write/thread path; `AgentGovernanceQueries` owns the read/analytics path (`agent_token_usage ⋈ agent_model_pricing`, tool calls, threads, runs). A store adapter implements both — the Lucid read-model reads the same tables the Lucid store writes.
</Callout>

## Wire it

```ts title="config/agent.ts"
import { defineConfig, stores, pricingStores, governanceQueries } from '@adonis-agora/agent'

export default defineConfig({
  model: () => aiSdkModel(openai('gpt-4o-mini')),
  store: 'lucid',
  stores: { lucid: stores.lucid() },
  pricingStore: pricingStores.lucid(),
  governanceQueries: governanceQueries.lucid(),
})
```

Configuring `governanceQueries` **mounts the `/agent/governance/*` read routes**. Omit it and they're never mounted (read-only surface; safe to leave off). The factory receives the already-resolved `pricingStore`, so the read-model prices its cost rollups against the *same* live prices the loop's cost fold uses. See [Quota & cost](/docs/agent/governance/quota-and-cost) for pricing.

## Who may read it

These routes serve **cross-actor** data — every actor's spend, usage, threads, runs, and pending approvals — so unlike the chat routes they are not owner-scoped. Instead they sit behind the [`governanceAuthorize`](/docs/agent/governance/authorization#governance-route-authorization) gate:

- Each route resolves the actor through the same resolver as the chat routes (`401` on failure), then runs `governanceAuthorize` (`403` on deny — the gate is **fail-closed**, so a throw denies too).
- **Omit `governanceAuthorize`** and any authenticated actor may read the whole read-model — the historical behavior, correct only when every authenticated caller is trusted staff. The provider logs a startup warning when the routes mount without a gate; typically you set it to an ADMIN check.

<Callout type="warn" title="Not open to any authenticated actor">
Earlier releases mounted these routes for any resolved actor. Since `0.5.0` the cross-actor read-model is gated by `governanceAuthorize`, and per-actor chat records are protected by [object-level ownership](/docs/agent/governance/authorization#object-level-ownership). Configure the gate (e.g. an ADMIN check) before exposing governance to a mixed audience.
</Callout>

## The routes

All read-only (`GET`), all authenticated **and** gated by `governanceAuthorize` (governance is cross-actor data). `from`/`to` are inclusive UTC days (`YYYY-MM-DD`); `limit` defaults to `50`, clamped to `200`.

| Method & path | Purpose |
| --- | --- |
| `GET /agent/governance/spend/model` | Per-model token + cost rollup over the range. |
| `GET /agent/governance/spend/actor` | Per-actor token + cost rollup over the range. |
| `GET /agent/governance/usage/trend` | The daily token + cost trend over the range. |
| `GET /agent/governance/tool-calls/recent` | Newest-first recent tool-call activity feed. |
| `GET /agent/governance/threads/recent` | Newest-first recent thread activity feed. |
| `GET /agent/governance/runs` | Filterable, cursor-paginated run list, newest-first. Query: `actor?`, `agent?`, `status?`, `from?`, `to?`, `cursor?`, `limit?`. |
| `GET /agent/governance/runs/:id` | One run's full trace — run + messages + tool calls + approvals + usage — or `null`. |
| `GET /agent/governance/approvals/pending` | Cross-thread HITL approvals inbox, oldest first. Query: `actor?`, `limit?`. |
| `GET /agent/governance/tools/stats` | Per-tool call/failure/rejection/latency rollup. Query: `from?`, `to?`. |
| `GET /agent/governance/reliability` | Success/failure/cancel rates + mean settled duration. Query: `from?`, `to?`. |

The matching `AgentGovernanceQueries` methods are `spendByModel`, `spendByActor`, `usageTrend`, `recentToolCalls`, `recentThreads`, `listRuns`, `runDetail`, `pendingApprovals`, `perToolStats`, and `runReliability`.

## The per-actor approvals inbox

There's one more route mounted alongside the read-model that is **not** behind `governanceAuthorize`:

| Method & path | Purpose |
| --- | --- |
| `GET /agent/approvals/mine` | The **calling actor's own** pending HITL approvals, oldest first. Query: `limit?` (default `50`, clamped to `200`). |

It reads the same store as the governance inbox — `pendingApprovals({ actor })` filters by the owning run's `actor_ref` — but is always scoped to `actor.id`, so it needs no admin gate. This is the route a non-admin surface (a coordinator's chat, an operator console) polls to discover *its own* suspended tool calls, even when the cross-actor `GET /agent/governance/approvals/pending` inbox above is ADMIN-only. The two are complementary: `approvals/mine` is per-actor and always available; `governance/approvals/pending` is platform-wide and governance-gated.

## Run lifecycle tracking

Both runners record each run (turn) to an `agent_run` table through `store.recordRunStart(...)` / `recordRunEnd(...)`, tracked through `running → completed | failed | cancelled` with `stepCount`, token totals, an `error`, and a `durable` flag (true under the [durable runner](/docs/agent/durability/durable-runner)). A run's messages, tool calls, and usage back-reference its `run_id`.

<Callout type="info" title="Run tracking is an additive migration">
`node ace configure @adonis-agora/agent` publishes a `create_agent_run_tracking` migration alongside the core tables — the `agent_run` table plus the `run_id` back-references. Run it after the base migration. Adapters backed by a store without run recording return an empty page / `null` / zeros from the run-lifecycle methods, so the surface degrades gracefully.
</Callout>

## Actor labels

Persisted `actorRef`s are opaque by design (no FK into your user table). To render names instead of raw refs on governance surfaces, bind an **actor directory** — the read-side dual of the [actor resolver](/docs/agent/governance/authorization):

```ts title="config/agent.ts"
import { actorDirectories } from '@adonis-agora/agent'

export default defineConfig({
  // ...
  actorDirectory: actorDirectories.memory({ labels: { u_1: 'Ada Lovelace' } }),
})
```

`ActorDirectory.resolveDisplay(refs)` returns a `ref → label` map; an unknown ref is simply omitted (surfaces fall back to the raw ref) and it never throws. Bind your own over the host's user table, or omit it entirely.

## In tests

`InMemoryGovernanceQueries` (with a settable `InMemoryModelPrice` table) implements the same SPI over in-memory data, so a test exercises the identical read-model code paths without a database. It's in the [testing kit](/docs/agent/testing).
